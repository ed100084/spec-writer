## 一、投標資格設計

### 1.1 三層架構法

| 層級 | 目的 | 法律依據 | 範例 |
|------|------|----------|------|
| **基本資格** | 法定最低門檻，無裁量空間 | 採購法第 102、103 條 | 公司登記、稅務證明、無退票、非拒絕往來 |
| **專業資格** | 確保廠商有履約能力 | 採購法第 52、54 條 | 近 3 年 2 件以上類似實績（金額 ≥ N 萬） |
| **評分項目** | 優於基本資格的加分項 | 同上 | ISO 認證、團隊證照、雲端能力 |

### 1.2 常見違法陷阱

```
❌ 「投標廠商須具備 ISO 27001 認證，否則廢標」
→ 若採購金額未達一定標準或該認證與履約無直接關聯，構成不合理限制。

✅ 「ISO/IEC 27001 認證納入技術評審評分項（加分），非投標門檻。」
```

### 1.3 實績驗證三齊備原則

| 文件 | 目的 | 審查重點 |
|------|------|----------|
| **合約影本** | 確認案號、金額、期間 | 雙方印章、簽約日期、品項明細 |
| **驗收證明** | 確認系統已交付且合格 | 客戶簽章、驗收日期、無保留條款 |
| **系統畫面** | 確認系統確實運作中 | ≥3 張核心功能截圖、含時間戳記或網址 |

> **查核機制**：規格書中應明訂「評審單位得對可疑實績進行到府查核或函查，發現虛偽不實依採購法第 101 條處理」。

## 二、需求規格撰寫模式

### 2.1 MoSCoW 必要性分類

| 分類 | 定義 | 驗收影響 |
|------|------|----------|
| **Must** | 系統必須具備，缺少即無法運作 | Fail（不予驗收） |
| **Should** | 重要但非關鍵，有替代方案 | Conditional Pass（限期補正） |
| **Could** | 期望功能，不影響核心營運 | 可放進第二期 |
| **Won't** | 本案明確排除 | Out-of-Scope |

### 2.2 需求表格標準格式

```markdown
| 編號 | 需求描述 | 使用角色 | 必要性 | 驗收方式 |
|------|----------|----------|--------|----------|
| FR-001 | 系統須支援多國語言切換（繁中/英文） | 一般使用者 | Must | TC-001: 切換後介面完整翻譯無亂碼 |
```

### 2.3 模糊用語清單（禁止使用）

| 類別 | 禁用詞 | 正確寫法 |
|------|--------|----------|
| 效能 | 「良好」「高效」 | 「Response Time ≤ 2 秒 (95th percentile)」 |
| 容量 | 「適當」「大量」 | 「支援 ≥ 10,000 筆/秒處理量」 |
| 時機 | 「必要時」「視情況」 | 「於登入後 5 分鐘內自動儲存」 |
| 功能 | 「可支援」「部分符合」 | 「須完整支援 X 功能，不得僅部分實作」 |

## 三、AI 模組規格

### 3.1 AI 需求檢查清單

| 領域 | 必須明確的項目 |
|------|----------------|
| **模型來源** | 自建 / 第三方 API / ��合；供應商名稱；部署位置（院內/雲端） |
| **資料流向** | 訓練資料來源、推論資料是否離院、是否境外傳輸、留存政策 |
| **準確率指標** | F1 Score / BLEU / WER / 醫師盲測通過率 — 依功能定義客觀指標 |
| **Hallucination 防護** | 原文比對機制、信心分數閾值、人工覆核流程 |
| **人工覆核** | 醫療紀錄/診斷建議不得未覆核即寫入正式病歷（100% 覆核） |
| **稽核紀錄** | prompt、input、output、模型版本、覆核人員與時間戳記 |

### 3.2 AI 風險分級

| 等級 | 範例 | 管控要求 |
|------|------|----------|
| **高風險** | 輔助診斷、病歷摘要生成 | 100% 人工覆核 + DPIA + Pen Test |
| **中風險** | 就診時間預測、候診管理 | 準確率指標 + Audit Log |
| **低風險** | 內部知識庫搜尋 | 模型來源揭露 |

## 四、資訊安全條款

### 4.1 資安控制項矩陣

| 控制項 | 要求 | 驗收證明 | 罰則 |
|--------|------|----------|------|
| **ISMS** | 遵守本院 ISMS，處理 PHI 需最小化 | 廠商切結 + DPIA 報告 | 未通過不得開工 |
| **弱點掃描** | Critical/High = 0；Medium 有修補計畫 | 掃描報告（含日期、工具） | 不予驗收 |
| **滲透測試** | 涉及病人資料之系統須完成 Pen Test | Pen Test 報告 + 補強證明 | 不予驗收 |
| **SBOM** | 提供 Software Bill of Materials | SPDX/CycloneDX 格式 | 要求補交 |
| **MFA** | 管理員帳號、遠端存取須啟用多因素認證 | 設定截圖/測試報告 | 限期改善 |
| **Audit Log** | 登入、權限變更、資料存取、API 呼叫皆紀錄，保存 ≥6 個月 | Log 範例 + 機制說明 | — |

### 4.2 資安事件通報流程

```
發現資安事件
    ↓
4 小時內通知院方資安室（電話 + Email）
    ↓
24 小時內提出初步報告（影響範圍、原因、處置）
    ↓
72 小時內提出完整報告（根因分析、補強措施）
    ↓
違約金計收（依合約金額 X%/事件）；情節重大者得終止契約
```

### 4.3 禁用項目強制清單

| # | 禁用項目 | 說明 |
|---|----------|------|
| D1 | 中資硬體/軟體/雲端 | 華為、中興、阿里雲、騰訊雲、百度雲等 |
| D2 | 中資 SDK/函式庫 | 第三方元件不得含中資背景程式碼 |
| D3 | 弱加密演算法 | MD5、SHA-1（簽章）、DES、RC4 |
| D4 | 預設帳號/密碼 | 交付時須全部變更 |

## 六、驗收標準設計

### 6.1 三級判定規則

| 判定 | 定義 | 處理方式 |
|------|------|----------|
| **Pass** | 所有 Must 項目達標，無 Critical/High 資安缺失 | 可驗收付款 |
| **Conditional Pass** | Must 無重大缺失，Should 有可補正缺失 | 限期（通常 30 日）補正後付款或保留款 |
| **Fail** | 任一 Must 未達標，或存在 Critical/High 資安缺失 | 不予驗收；廠商須改善後重新驗收 |

### 6.2 測試案例表格範例

```markdown
| 編號 | 對應需求 | 測試資料 | 步驟 | 預期結果 | 判定 |
|------|----------|----------|------|----------|------|
| TC-001 | FR-001 | 測試帳號 A（角色：醫師） | 1. 登入 → 2. 填寫同意書 → 3. 簽署 | 系統儲存成功，產生 PDF 並可下載 | Pass/Fail |
```

### 6.3 付款節點對應表

| 節點 | 百分比 | 前置條件 |
|------|--------|----------|
| 簽約金 | X% | 合約簽訂 |
| 初驗款 | X% | UAT 通過，提交初驗報告 |
| 驗收款 | X% | Final Acceptance 完成，所有文件交付 |
| 保固尾款 | X% | 保固期滿無未處理事項 |

> **注意**：付款節點須與驗收節點一致，不得有「驗收未完成但需付款」之模糊空間。

## 七、SLA 與維運

### 7.1 SLA 分級標準

| 等級 | 定義 | 回應時間 | 修復時間 | 通報方式 |
|------|------|----------|----------|----------|
| **Critical** | 系統中斷、醫療作業受阻、資安事件 | ≤ 1 小時 | ≤ 4 小時 | 電話 + Email |
| **Major** | 主要功能異常但有替代流程 | ≤ 4 小時 | ≤ 1 工作日 | Email / 工單 |
| **Minor** | 不影響主要作業之瑕疵 | ≤ 1 工作日 | ≤ 5 工作日 | 工單 |

### 7.2 SLA 未達標罰則範例

```markdown
| 月份 | SLA 實際達成率 | 約定目標 | 違約金計算 |
|------|---------------|----------|------------|
| 2026-07 | 98.5% | ≥ 99.5% | 扣減當月維運費用之 X% |
```

## 八、智財權與授權

### 8.1 智財權歸屬矩陣

| 類型 | 歸屬 | 交付物 |
|------|------|--------|
| **客製開發程式碼** | 甲方（院方）所有 | 完整原始碼 + 編譯工具鏈 + 文件 |
| **COTS 既有產品** | 廠商所有，授予甲方使用授權 | 授權憑證 + 安裝檔 |
| **第三方開源元件** | 依各元件授權條款 | SBOM + 授權文本 |

### 8.2 BOM 明細欄位說明

| 欄位 | 填寫範例 | 說明 |
|------|----------|------|
| 品項名稱 | 「同意書管理系統 — Enterprise Edition v3.2」 | 完整名稱 + 版本號 |
| 授權類型 | 「永久 / 年訂 / SaaS / per-seat」 | 不得僅寫「軟體授權」 |
| 授權數量 | 「100 user / unlimited device」 | 明確單位 |
| 授權期間 | 「2026-07-01 ~ 2027-06-30」或「永久」 | 訂閱制須註明起迄 |
| 單價/總價 | NT$ 500,000 / NT$ 500,000 | 含稅/不含稅需標示 |

### 8.3 禁止版本清單

```
❌ Express / Community / Trial / Developer / Evaluation 版本
✅ Commercial / Standard / Enterprise 版本
```

## 九、Data Portability / 退場機制

### 9.1 Data Portability 要求矩陣

| 項目 | 要求 |
|------|------|
| **觸發時點** | 合約終止、系統汰換、院方要求時 |
| **交接期限** | 院方通知後 30 日曆天內 |
| **交接格式** | FHIR R4 / CSV / JSON / SQL Dump / 雙方議定開放格式 |
| **交接內容** | 主資料、交易資料、附件、Audit Log、AI output、設定檔 |
| **驗證方式** | 院方抽測匯入、筆數核對、Hash 驗證 |

### 9.2 禁止事項（強制條款）

```markdown
廠商不得以以下理由拒絕或延遲交接院方資料：
- 合約終止或費用爭議
- 商業機密保護
- 第三方授權限制（廠商須事先確保可轉移性）
```

## 📝 撰寫原則速查

| 原則 | 說明 | 範例 |
|------|------|------|
| **明確動作詞** | 使用「須」「應」而非「應該」「可以」 | ✅ 「廠商須完成系統安裝」<br>❌ 「系統應該有安裝功能」 |
| **量化指標** | 所有需求須可測量 | ✅ 「≤ 2 秒 (95th percentile)」<br>❌ 「快速回應」 |
| **唯一編號** | 每項需求有獨立 ID | FR-001, NFR-002, INT-003 |
| **驗收對應** | Must 需求須有測試案例 | TC-XXX → FR-XXX |
| **Fail Loud** | 資安缺失直接 Fail，不給 Conditional Pass | Critical/High = 0 |
